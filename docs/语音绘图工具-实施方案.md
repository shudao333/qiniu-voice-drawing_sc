## 0. 一句话目标

做一个**纯语音控制的网页绘图工具**:用户不碰鼠标键盘,只靠说话完成"画圆、改颜色、移动、删除、撤销、组合图形"等创作。评委关注三件事:
1. **指令理解的准确性与容错性**
2. **语音到绘图操作的响应延迟**
3. **复杂指令的拆解与执行能力**

方案的每一个设计决策都围绕这三点展开。

---

## 1. 赛题拆解与设计取向

| 评分维度 | 我们的应对策略 |
| --- | --- |
| 指令准确性 | 两级解析:本地规则(关键词/正则)先命中常见指令,LLM 兜底复杂/模糊指令 |
| 容错性 | 同义词归一 + 澄清式反问("你是说红色的圆吗?")+ 失败不崩溃 |
| 响应延迟 | 简单指令走本地解析(<100ms),不必每句都请求云端;流式语音识别边说边出文字 |
| 复杂指令拆解 | LLM 输出"指令序列"(数组),前端按序执行;支持组合图形("画一个房子") |
| 端云协同/成本 | 本地能处理的不上云;LLM 只在必要时调用 —— 这正好是设计文档的加分点 |

**核心设计哲学:不是每句话都丢给大模型。** 这既降延迟又降成本,是这个赛题最重要的工程判断,务必写进设计文档。

## 2. 技术选型

### 2.1 为什么用 Web 前端而不是纯 Python 桌面

绘图工具天然适合浏览器:Canvas 绘图成熟、`Web Speech API` 浏览器原生支持麦克风录音和语音识别、部署只需开一个静态页 + 一个轻后端。你说"对 JS 不太确定"没关系——本方案的 JS 部分都是**框架无关的原生写法**(或极简 React),后端用你最熟的 **Python**,衔接点只有一个 HTTP 接口,心智负担很低。

### 2.2 推荐技术栈

```
前端 (/frontend)
  ├─ 原生 HTML + CSS + JavaScript(不强制上框架,降低你的 JS 心智负担)
  ├─ Konva.js          —— Canvas 绘图库,自带图形对象/选中/拖拽,比裸 Canvas 省一半代码
  ├─ Web Speech API    —— 浏览器原生语音识别(SpeechRecognition),零成本、低延迟、做兜底
  └─ MediaRecorder     —— 录音上传给云端 ASR(当需要更高识别准确率时)

后端 (/backend)
  ├─ Python 3.10+ + FastAPI    —— 你最熟的语言,FastAPI 自带交互式文档便于演示
  ├─ uvicorn                   —— ASGI 服务
  ├─ httpx / openai SDK        —— 调七牛云大模型(OpenAI 兼容)
  └─ python-dotenv             —— 管理 API Key,绝不进仓库
```

### 2.3 七牛云平台接入(已核实)

- **Base URL**: `https://api.qnaigc.com/v1`,**完全兼容 OpenAI 格式**
- 文本对话:`POST /v1/chat/completions`,示例模型 `deepseek-v3`(也支持 50+ 主流模型,用 `GET /v1/models` 查可用列表)
- 平台官方说明其提供 **ASR/TTS + 多模态**能力
- API Key 在 `portal.qiniu.com` 控制台获取,比赛通常配套免费 Token 额度
- 因为 OpenAI 兼容,Python 端可直接用 `openai` 库:
  ```python
  from openai import OpenAI
  client = OpenAI(base_url="https://api.qnaigc.com/v1", api_key=os.environ["QINIU_API_KEY"])
  ```

### 2.4 语音识别(ASR)双通道策略 —— 重要的容错/成本设计

| 通道 | 用途 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **主**:浏览器 Web Speech API | 实时把语音转文字 | 零成本、零延迟、边说边出 | Chrome/Edge 支持好,需联网,识别中文一般够用 |
| **备**:云端 ASR(七牛云/录音上传) | Web Speech 不可用或识别差时 | 准确率高 | 有网络往返延迟、消耗额度 |

**先用浏览器原生,做不到再上云**——这是端云协同的核心体现,务必写进设计文档。
如果浏览器环境识别中文效果不理想,再切到云端 ASR 录音上传方案。

> 注意:`SpeechRecognition` 在 Chrome/Edge 下走的是 Google 后端,**国内网络可能不稳**。开题第一天的 PR2 就要实测,若不可用则把云端 ASR 提到主通道。这条"风险与决策"也是设计文档的好素材。

### 3.1 指令的统一数据结构(贯穿前后端的契约)

整个项目的关键抽象是这个 JSON —— LLM 输出它、前端执行它、本地规则也生成它:

```json
{
  "commands": [
    { "action": "draw", "shape": "circle", "props": { "x": 300, "y": 200, "radius": 50, "color": "#ff0000" } },
    { "action": "draw", "shape": "rect",   "props": { "x": 100, "y": 100, "width": 120, "height": 80, "color": "#0000ff" } },
    { "action": "modify", "target": "last", "props": { "color": "#00ff00" } },
    { "action": "move",   "target": "selected", "props": { "dx": 50, "dy": 0 } },
    { "action": "delete", "target": "last" },
    { "action": "undo" }
  ],
  "reply": "好的,我画了一个红色的圆和一个蓝色的方块"
}
```

支持的 `action`:`draw / modify / move / delete / undo / redo / clear / select / group`。
`target` 可为:`last`(最近)、`selected`(当前选中)、`all`、或具体描述("那个红色的圆")。
**先把这个 schema 定死,前后端围绕它独立开发——这是项目能两天做完的关键。**

### 4.3 是否需要 Node?——给你一个判断

- **不想碰 JS 工具链** → 前端就是 `index.html + app.js + style.css` 三个文件,用 `python -m http.server` 或让 FastAPI 直接托管静态文件。Konva 用 CDN 引入。**推荐你走这条,零额外环境。**

> 结论:**先不装 Node,纯静态前端 + FastAPI 托管**。整个项目只依赖 Python 环境 + 一个 CDN。

### PR4 · 指令契约 `feat/command-schema`  〔Day 1 晚〕
- **做什么**:用 Pydantic 在后端定义指令结构(第 3.1 节),前端写一份对应的 JSDoc/常量。给出所有 `action` 的枚举。
- **实现思路**:这是前后端共同遵守的契约,定死它,双方可并行开发。
- **测试方式**:后端 `pytest` 跑 schema 校验用例(合法/非法各一);`GET /api/schema` 返回结构说明。
- **Skill**:`Planning with Files` 记录契约定义。

### PR5 · 本地规则解析 `feat/local-parser`  〔Day 1 晚 → Day 2 上午〕
- **做什么**:纯前端 JS,用关键词/正则把常见中文指令转成指令对象。覆盖:形状(圆/方/矩形/线/三角/文字)、颜色(红黄蓝绿黑白…中文色名→hex)、动作(画/删除/撤销/清空)。
- **实现思路**:维护"同义词词典"(圆=圆形=圈,删掉=删除=去掉),正则提取数量/颜色;命中即返回指令,不命中返回 null(交给 LLM)。**这是低延迟的核心**。
- **测试方式**:`parser.test` —— 输入"画一个红色的圆"→ 输出 `{action:'draw',shape:'circle',props:{color:'#ff0000'}}`,跑 20 条样例。
- **Skill**:`Code Review`(自审解析逻辑的边界 case)。

### PR6 · 执行引擎 `feat/executor`  〔Day 2 上午〕
- **做什么**:`executeCommands(commands[])`,把指令序列逐条落到 Konva 画布。先实现 `draw`:根据 shape+props 在合理位置画出图形并记入 `shapes[]`。
- **实现思路**:每个 action 一个 handler 函数(策略模式);draw 时若没给坐标就用画布中心 + 轻微随机偏移避免重叠。
- **测试方式**:把 PR5 解析出的指令喂进来,画布出现对应图形;连续画两个不重叠。
- **Skill**:`Code Simplifier` —— 让 handler 分发保持干净。
- 到这里:**说"画一个红色的圆"→ 真的画出来了**,核心闭环跑通(本地链路)。这是一个里程碑 PR。

### PR7 · LLM 解析后端 `feat/llm-parse`  〔Day 2 下午〕
- **做什么**:`POST /api/parse {text}`,后端调七牛云 LLM,用精心设计的 system prompt 让模型**只输出第 3.1 节的指令 JSON**,后端用 Pydantic 校验后返回。
- **实现思路**:
  - `base_url=https://api.qnaigc.com/v1`,OpenAI 兼容,用 `openai` 库
  - system prompt 里塞入 schema 定义 + few-shot 示例 + "只返回 JSON" 约束;可用 `response_format={"type":"json_object"}` 强制 JSON
  - 模型先用 `deepseek-v3`,跑通后可比较其它模型效果/延迟
  - 校验失败时让模型重试一次或降级
- **测试方式**:`curl -X POST .../api/parse -d '{"text":"画一个房子,门是棕色的"}'`,返回结构化指令序列(房子 = 矩形墙 + 三角顶 + 矩形门)。
- **Skill**:`Code Review`(强制,审 prompt 注入风险与 JSON 解析健壮性);`MCP Builder` 可作封装外部模型调用的参考。
- ⚠️ 安全:`text` 来自用户语音,prompt 里要做好边界,避免越权指令;Key 从环境变量读。

### PR8 · 解析路由 `feat/parser-router`  〔Day 2 下午〕
- **做什么**:前端统一入口 `parse(text)`:先走 PR5 本地解析,命中直接执行(快);未命中才 `await api.parse(text)` 走 LLM。
- **实现思路**:本地优先 = 低延迟 + 省 Token;LLM 兜底 = 高覆盖。加载态提示"思考中…"。
- **测试方式**:说"画个圆"瞬间出图(本地);说"画一只可爱的小猫的脸"走 LLM 返回多指令。对比两者延迟并记录(写进 DESIGN)。
- **Skill**:`Code Review`。
- 这是**端云协同/成本控制**最直接的体现,演示和文档都要重点讲。

### PR9 · 修改/移动/选中 `feat/modify-move`  〔Day 2 晚〕
- **做什么**:实现 `modify`(改颜色/大小)、`move`(上下左右移动)、`select`(按描述或"最近的"选中并高亮)。
- **实现思路**:`target` 解析成具体图形对象(last/selected/按颜色形状匹配);选中加描边高亮。
- **测试方式**:画圆后说"把它变成蓝色"→ 变蓝;"向右移动一点"→ 右移;"选中那个红色方块"→ 高亮。
- **Skill**:无需(逻辑直接)。

### PR10 · 撤销/删除 `feat/undo-delete`  〔Day 2 晚〕
- **做什么**:`undo/redo/delete/clear`。维护操作历史栈。
- **实现思路**:每次改变画布把"逆操作"压栈;undo 弹栈执行逆操作;redo 对称。delete 移除目标图形。
- **测试方式**:画三个图 →"撤销"逐个消失 →"重做"恢复;"删除最后一个"/"清空画布"生效。
- **Skill**:`Code Review`(撤销栈最容易出 bug,重点审)。

### PR11 · 容错与澄清 `feat/error-tolerance`  〔Day 2 晚 → Day 3 上午〕
- **做什么**:三层容错——① 同义词/口语归一("搞个""来个"="画一个");② 识别不全时澄清式反问("你想画什么形状?");③ 任何解析失败都给友好提示,绝不崩。
- **实现思路**:本地词典扩充 + LLM prompt 增加"信息不足时返回 `clarify` 动作 + 追问文字";前端遇到 `clarify` 就用语音/文字问回去,进入"等待补充"状态。
- **测试方式**:说"画那个"(信息不足)→ 系统反问;说带口音/含糊的指令仍能合理处理;说乱码不崩溃。
- **Skill**:`Code Review`(强制)——容错是评分维度,重点审边界。
- 这是**指令准确性与容错性**评分维度的主战场,要做扎实并写进 DESIGN。

### PR12 · 复杂指令拆解 `feat/complex-commands`  〔Day 3 上午〕
- **做什么**:支持一句话多步("画一个红圆,旁边画一个蓝方块,然后把圆移到上面")和组合图形("画一个房子/笑脸/小车")。
- **实现思路**:LLM 已能输出指令序列(数组),executor 已能按序执行——本 PR 主要是**强化 prompt 的拆解能力 + 加组合图形的 few-shot 模板**,并保证多指令按序、引用关系("把圆"指前面画的圆)正确。
- **测试方式**:说"画一个房子"→ 出现墙+顶+门多个图形;说"画三个从小到大的圆"→ 三个递增圆。
- **Skill**:`Code Review`。
- 这是**复杂指令拆解能力**评分维度,是拉开差距的亮点,demo 必演。

### PR13 · 语音反馈 `feat/voice-feedback`  〔Day 3 下午〕
- **做什么**:操作完成后给反馈——优先文字气泡;进阶用 `SpeechSynthesis`(浏览器原生 TTS)或七牛云 TTS 读出 `reply`。
- **实现思路**:执行完指令把 `reply` 文本显示+朗读;形成"说话→看到画→听到确认"的完整语音交互闭环。
- **测试方式**:说"画个圆",听到"好的,画了一个圆"。
- **Skill**:无需。
- 加分点:让工具有"对话感",契合赛题"纯语音交互"定位。

### PR14 · 界面美化 `feat/ui-polish`  〔Day 3 下午〕
- **做什么**:整体视觉打磨——麦克风状态动效(聆听中波纹)、实时识别字幕条、可折叠的"语音指令帮助"面板(列出支持的说法)、画布工具感。
- **实现思路**:纯 CSS 动效 + 清晰的状态机视觉(待命/聆听/思考/执行)。指令帮助面板对评委理解能力边界很重要。
- **测试方式**:各状态切换视觉正确;新人不看文档也知道能说什么。
- **Skill**:`UI UX Pro Max`(强制)——这个 PR 就是交给它发挥的。

### PR15 · 文档收尾 `docs/design-readme`  〔Day 3 晚,截止前留足缓冲〕
- **做什么**:
  - **DESIGN.md**(赛题强制要求):计划支持哪些指令能力 / 最终实现了哪些 / 未完成部分及原因;成本控制技巧(本地优先、缓存、模型选择)。
  - **README.md**:项目简介、技术栈、依赖清单(列明 Konva/FastAPI/七牛云等第三方,标注原创部分)、安装运行步骤、**demo 视频链接**、指令示例。
  - 录 **demo 视频**:全程语音讲解,覆盖核心模块(语音输入→简单指令→复杂指令→容错→撤销),传 B 站/云盘,链接放 README。
- **测试方式**:照 README 从零跑通;视频链接可播放。
- **Skill**:`Planning with Files`(对照最初任务清单核对完成度,生成"已完成/未完成"对照表,直接喂给 DESIGN.md)。

## 7. Skill 使用策略(给你的 coding agent 的规则)

你装的这些 Skill,建议这样用——分为"全程""按 PR 类型强制""按需推荐"三档:

### 7.1 全程贯穿
- **Planning with Files**:开题就让 agent 用它把本方案落成可勾选的任务清单文件(如 `TASKS.md`),每完成一个 PR 勾一项。它也是 PR15 生成"完成度对照表"的工具。
- **Superpowers**:作为 agent 的通用能力底座,平时让它自行调度即可,不必每次手动点。

### 7.2 按 PR 类型强制挂载

| PR 类型 | 强制 Skill | 原因 |
| --- | --- | --- |
| 任何动到界面的(PR3/14) | **UI UX Pro Max** | 视觉/交互是评委直观感受,交给专精 skill |
| 涉及解析正确性/容错(PR5/7/8/11/12) | **Code Review** | 这几个直接对应评分维度,必须严审边界 |
| 涉及 prompt/外部模型调用(PR7) | **Code Review** + 参考 **MCP Builder** | 防 prompt 注入、保证 JSON 健壮 |

### 7.3 按需推荐(让 agent 自己判断或你手动 @)
- **Code Simplifier**:某个文件变长变乱时(尤其 executor/parser),@它重构。
- **Webapp Testing**:PR2/14 等需要真在浏览器里验证麦克风、交互流程时用。
- **Code Review 全家桶**(code-reviewer / code-review-excellence / requesting-code-review):**每个 PR 合并前过一遍**做自审,这正好对应比赛"PR 要清晰、合并后可运行"的要求。
- **Skill Creator / MCP Builder**:本项目用不上造新 skill;MCP Builder 仅作"如何优雅封装外部 API"的参考读物。
- **Ralph Loop**:适合"反复迭代直到通过测试"的场景。可用于 PR5/PR7 这类需要反复调样例直到准确率达标的解析任务——让它循环跑测试样例并改进,直到通过率达标。**但要盯着别让它空转烧 Token。**

### 7.4 一个实操建议
在仓库根放一个 `AGENTS.md` 或 `.agent/rules`,写明:"每个 feature PR 完成后必须调用 Code Review skill 自审;动 UI 的 PR 必须先咨询 UI UX Pro Max。" 这样 agent 每次会自动遵守,不用你每次提醒。

> 比赛硬规则:commit 时间戳必须落在 06-12 00:00 ～ 06-14 23:59;**只在最后一天导入全部代码 = 无效**。所以每天都要有真实 commit 和 PR。

## 9. 指令能力清单(给 DESIGN.md 用,也是开发对照表)

赛题要求 DESIGN.md 写"计划支持哪些 / 实现了哪些 / 未完成原因"。下面是建议的能力清单,开发时逐项打勾:

**基础绘图**
- [ ] 画圆 / 矩形 / 直线 / 三角形 / 文字
- [ ] 指定颜色(中文色名 → 十余种)
- [ ] 指定大小(大/中/小、具体数值)
- [ ] 指定位置(中间/左上/具体坐标/"在旁边")

**编辑操作**
- [ ] 修改颜色 / 大小
- [ ] 移动(方向 + 距离)
- [ ] 选中(按描述/最近的)
- [ ] 删除 / 清空
- [ ] 撤销 / 重做

**容错能力**
- [ ] 口语同义词归一
- [ ] 信息不足时澄清式反问
- [ ] 无法识别时友好降级、不崩溃

**复杂指令**
- [ ] 一句话多步操作
- [ ] 组合图形(房子/笑脸/小车等)
- [ ] 指令间引用("把刚才那个圆…")

**语音交互**
- [ ] 实时语音转文字
- [ ] 操作结果语音/文字反馈

> 标★的(复杂指令、容错)是评分重点,优先保证;时间不够时,语音反馈 TTS、redo 等可作为"未完成项"诚实写进文档(诚实说明未完成原因也是规则鼓励的)。

## 10. 成本控制 / 端云协同(DESIGN.md 加分项)

赛题没把成本列为本题硬指标,但"语音到操作的响应延迟"和工程成熟度都和它相关,值得在 DESIGN 里专门写一段:

1. **本地优先解析**:80% 的常见指令(画圆、改色、撤销)用本地规则零成本零延迟解决,不调云端。
2. **LLM 仅兜底**:只有本地解析失败的复杂/模糊指令才请求大模型,大幅减少 Token 消耗。
3. **浏览器原生 ASR/TTS 优先**:Web Speech / SpeechSynthesis 不花钱,云端语音仅作备选。
4. **Prompt 精简 + 结构化输出**:system prompt 用紧凑 schema,强制 JSON 输出避免冗长生成,减少输出 Token。
5. **结果缓存**:相同文本指令缓存解析结果,重复指令不重复请求。
6. **模型选择**:简单解析任务用便宜/快的模型(如 deepseek-v3),不盲目上最贵模型。

把"计划用了哪些 / 实际用了哪些"列成两栏写进 DESIGN——这正是赛题问的。

## 11. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| Web Speech API 国内网络不稳/不可用 | 语音输入是命根子 | **D1 立刻实测**,不行就切云端 ASR 录音上传方案(PR2 备选) |
| 七牛云 Key 申请慢/额度有限 | 阻塞 LLM 链路 | D1 先申请;本地解析先行,LLM 链路解耦,没 Key 也能演示基础功能 |
| LLM 输出不合法 JSON | 解析崩溃 | 强制 `json_object` + Pydantic 校验 + 失败重试/降级 |
| 中文语音识别准确率低 | 指令解析错 | 同义词归一 + 澄清反问 + 关键词模糊匹配兜底 |
| 时间不够 | 功能砍 | 严格按里程碑;复杂指令/容错优先,TTS/redo 可砍并诚实写进文档 |
| 误把密钥提交 | 安全 + 可能判违规 | `.gitignore` 先行;提交前 `git diff` 自查;万一泄露立即重置 Key |
