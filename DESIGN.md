# 设计文档 · AI 语音绘图工具 (Voice Draw)

> 七牛云 × XEngineer 暑期实训营 · 题目二：AI 语音绘图工具
> 本文档说明：计划支持的能力、最终实现情况、未完成项及原因、成本与延迟控制、系统架构、关键设计决策与权衡。

---

## 一、一句话目标

做一个**纯语音控制的网页绘图工具**：用户不碰鼠标键盘，只靠说话完成「画圆、改颜色、移动、删除、撤销、画组合图形」等创作。

围绕赛题三大评分维度展开设计：
1. **指令理解的准确性与容错性**
2. **语音到绘图操作的响应延迟**
3. **复杂指令的拆解与执行能力**

---

## 二、核心设计哲学

**不是每句话都丢给大模型。**

系统采用**两级解析架构**：常见指令（画圆、改色、撤销、清空）由前端本地规则解析器零延迟命中；只有复杂、模糊、组合类指令才升级到云端大模型。这既降低延迟，又减少 Token 消耗，是本项目最重要的工程判断。

## 四、未完成项及原因

| 未完成项 | 原因 | 备注 |
| --- | --- | --- |
| 云端 ASR 备用通道 | Web Speech API 在测试环境（Chrome/Edge）下中文识别可用，未触发切换云端的必要性 | 架构已为双通道预留，主通道够用故未实现备用；见风险与权衡 §七 |

| 写实级图像生成 | 本项目定位为「矢量基础图元组合」的简笔/卡通绘图，写实需文生图模型，偏离赛题"语音控制矢量绘图"主线 | 见风险与权衡 §七 的定位选择 |

> 诚实说明：以上未完成项均为**主动取舍**而非能力缺失。核心评分维度（准确性/容错、延迟、复杂指令拆解）已全部覆盖。

### 关键数据结构（前后端契约）

整个项目的核心抽象是统一的指令 JSON —— LLM 输出它、前端执行它、本地规则也生成它：

```json
{
  "commands": [
    { "action": "draw", "shape": "circle", "props": { "x": 300, "y": 200, "radius": 50, "color": "#ff0000" } },
    { "action": "modify", "target": "last", "props": { "color": "#00ff00" } },
    { "action": "move", "target": "#3", "props": { "dx": 50, "dy": 0 } }
  ],
  "reply": "好的，我画了一个红色的圆"
}
```

- 支持的 `action`：`draw / modify / move / delete / undo / redo / clear / select / group / clarify`
- 支持的 `shape`：`circle / rect / line / triangle / text / ellipse / polygon / arc / wedge`
- 支持的 `target`：`last`（最近）、`selected`（当前选中）、`all`、`#N`（精确索引）、或具体描述（"红色的圆"）

## 六、成本 / 延迟控制设计

| 设计 | 计划 | 实际采用 | 效果 |
| --- | :---: | :---: | --- |
| 本地优先解析 | ✅ | ✅ | 常见指令（画圆/改色/撤销/清空/移动）本地正则零延迟命中，不上云 |
| LLM 仅兜底 | ✅ | ✅ | 仅复杂/模糊指令请求大模型，大幅减少 Token 消耗 |
| 浏览器原生 ASR/TTS | ✅ | ✅ | Web Speech / SpeechSynthesis 零成本，不调云端语音接口 |
| 强制结构化 JSON 输出 | ✅ | ✅ | `response_format=json_object` + 紧凑 schema，避免冗长生成 |
| 低温度采样 | ✅ | ✅ | temperature=0.1 保证指令格式稳定，减少重试 |
| 长句拦截分流 | ✅ | ✅ | 本地解析器对超长句/复杂连词主动降级 LLM，避免误判 |
| 画布状态上下文 | — | ✅ | 把画布现状随请求发给 LLM，提升指代准确率，减少澄清往返 |

**端云协同体现**：约 80% 的常见操作走本地（零延迟零成本），仅复杂创作走云端。这是延迟与成本控制的核心。

## 七、风险与权衡（关键设计决策）

### 1. Web Speech API 的网络依赖
- **风险**：`SpeechRecognition` 在 Chrome/Edge 下依赖联网识别，国内网络可能不稳。
- **决策**：主通道用浏览器原生（零成本、低延迟、实测可用），架构预留云端 ASR 备用通道但未实现（主通道够用）。这是「先用免费方案，不够再上云」的成本优先取舍。

### 2. 矢量图元组合 vs 文生图
- **权衡**：本工具定位「语音控制矢量绘图」，用基础图元组合出简笔/卡通画，而非写实图像。
- **理由**：赛题考的是「指令理解 + 拆解 + 执行」，评委想看「说一句话 → 系统拆成 N 个部件正确摆位渲染」，而非绘画艺术水平。矢量方案响应快、可编辑（能改色/移动/撤销），契合「绘图工具」定位；文生图无法二次编辑，偏离主线。

### 3. rect 坐标语义统一
- **问题**：早期前端把 rect 的 x,y 当中心点（减半宽高），但 LLM 默认当左上角，导致矩形位置偏移。
- **决策**：统一为「左上角」语义，并在 LLM 的 system prompt 中明确声明，保证前后端一致。

### 4. 组合图形整体操作的寻址机制
- **问题**：组合图形含重复部件（两只眼睛、两条腿都是"黑色的圆/线"），用「颜色+形状」描述定位时会反复命中同一个图形，导致"把人往上移"只移动一部分。
- **决策**：引入 **`#N` 精确索引寻址**。画布状态本就给 LLM 发送了每个图形的唯一序号，LLM 对整组操作时生成 `#1`~`#N` 指令，每个序号唯一，彻底避免重复命中。这是通用寻址原语，适用于任意组合图形与 move/modify/delete。

### 5. 本地解析与"把它"指代的协同
- **问题**：本地解析器无状态，"把它移动"中的"它"无法判断指向选中还是最近图形。
- **决策**：引入智能 `it` 目标类型——本地解析命中"把它"类指令时返回 `target:'it'`，执行引擎解析为「优先选中的，否则最近的」；含形状描述词（"把圆变绿"）则降级 LLM 结合画布状态精确处理。

### 6. 流式作画的动效选型
- **权衡**：图元逐个显现可用「缩放弹入」或「淡入」。
- **决策**：选淡入。因 line/polygon 等图元的缩放原点会导致图形飞走错位，而 opacity 淡入对所有图元类型都安全稳定。
